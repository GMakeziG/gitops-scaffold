## Current project capabilities

```sh
gitops-scaffold analyze docker-compose.yml \
  --format json \
  --output report.json

gitops-scaffold generate report.json \
  --app audiobookshelf \
  --namespace apps \
  --output ./generated/audiobookshelf

gitops-scaffold validate ./generated/audiobookshelf
```

Direct generation from Compose (no separate cached-report step) is also
supported:

```sh
gitops-scaffold generate docker-compose.yml \
  --app audiobookshelf \
  --namespace apps \
  --output ./generated/audiobookshelf
```

Both input paths (a `.json` report from `analyze --output`, or a Compose
file parsed fresh) converge on the exact same normalized
`(ApplicationDefinition, AnalysisResult)` pair and the same
`GenerationPipeline` — verified byte-identical via `diff -ru` during v0.3
development, so there is no separate/divergent code path to keep in sync.

## Important design decisions

- The tool generates maintainable GitOps scaffolding, not automatically
  production-ready deployments.
- Readability, determinism, safety, and human review take priority over
  generating the fewest lines of YAML.
- Docker Compose host ports do not become Kubernetes target ports —
  `host_port` has no ClusterIP-Service equivalent and never appears in any
  generated manifest field (it may appear only in README's
  `kubectl port-forward` example, as the arbitrary local side).
- Secret values are never written to generated manifests or reports —
  secret/non-secret classification is derived from `AnalysisResult.findings`,
  never re-derived from `EnvVar.value` at generation time (so a redacted
  cached report can't leak `"***REDACTED***"` into a ConfigMap).
- `secret.example.yaml` is documentation only (placeholder value
  `"CHANGE_ME"`) and is never added to `kustomization.yaml`.
- Required Secret references remain required — `secretKeyRef.optional` is
  only `true` when the original Compose variable was a shell-passthrough
  (no value at all); it's never blanket-set to allow startup.
- Compose healthchecks map to `readinessProbe` by default.
- Liveness probes are opt-in (`ScaffoldSettings.enable_liveness_probe`,
  default `False`) — never auto-duplicated from the readiness probe.
- `startupProbe` is used when `start_period` can be mapped safely (computed
  `failureThreshold`), instead of folding it into `initialDelaySeconds`.
- PVC sizes use configured defaults (`ScaffoldSettings.default_pvc_size`)
  and remain `REVIEW REQUIRED` because Compose has no storage-size model.
- Unknown UID/GID values are not invented — `securityContext.runAsUser` is
  only emitted when a UID was actually resolved; otherwise a
  `REVIEW REQUIRED` comment appears instead.
- Foreign files (present on disk but not proven to be gitops-scaffold's own
  prior output, per the `generation-report.json` ledger) are never
  overwritten, even with `--force`.
- v0.3.0 is the first complete end-to-end generation milestone —
  `analyze` → `generate` → `validate` all work together for real Compose
  input, not stubs.

## Known limitations at v0.3.0

Verified against current code and `docs/roadmap.md`/`docs/generation.md`:

- no Helm generation (explicitly out of scope through v1.0)
- no remote container registry / OCI image inspection (planned for v0.6)
- no GitHub repository inspection (planned for v0.7; only a registry stub
  parser exists — `parse()` raises `NotImplementedError`)
- no Dockerfile parser (planned for v0.6; only a registry stub exists)
- no automatic ExternalSecret/OpenBao generation (deferred; `generate`
  only ever produces `secret.example.yaml` placeholders)
- no Cloudflare Tunnel generation (deferred)
- no live `kubectl apply` (`validate --kubectl` only runs
  `kubectl kustomize` as a build check, never applies anything)
- no Flux reconciliation (deferred; output is plain Kustomize-compatible
  manifests, no `Kustomization`/`GitRepository` Flux CRDs generated)
- no automatic storage sizing (PVC size is always a configured default,
  always flagged `REVIEW REQUIRED`)
- no guaranteed application-specific health endpoint discovery (probes are
  translated only from a healthcheck Compose already declares; nothing is
  inferred when none exists — a `TODO`/`REVIEW REQUIRED` is emitted instead)
- no automatic secure non-root UID/GID selection (never invented; always a
  review item when Compose doesn't declare `user:` with a resolvable UID)
- no production-readiness guarantee — explicitly documented project
  philosophy: this generates maintainable scaffolding for human review, not
  a "deploy and forget" pipeline
- (also worth noting, not in the requested list) `build:`-context Compose
  services and Compose port ranges remain unsupported since v0.2; tmpfs
  mounts are correctly excluded from PVC conversion but no `emptyDir`
  alternative is wired in; only 2 of 6 originally-scoped scenarios
  (Audiobookshelf, multi-service) have full checked-in golden-tree fixtures,
  the rest are covered by targeted unit/integration tests instead

## Recommended next milestone

Do not begin implementation — this is a recommendation only.

Four directions were on the table:

1. **Security and policy analysis** — pod security recommendations,
   runtime-user inspection, privileged/host-path safeguards, seccomp and
   capability recommendations, a policy report.
2. **GitOps platform profiles** — generic/homelab profiles, Flux/Kustomize
   conventions, Traefik + cert-manager options, External Secrets/OpenBao
   templates.
3. **Container-image inspection** — OCI image metadata (`USER`, `EXPOSE`,
   `HEALTHCHECK`, `VOLUME`, `ENTRYPOINT`, `CMD`, labels).
4. **Interactive review workflow** — analyze → review unresolved decisions
   → save overrides → regenerate.

**Recommendation: (1) Security and policy analysis**, as the v0.4
milestone.

Reasoning:
- It builds directly on infrastructure that already exists and is proven
  out in v0.2/v0.3 — the `DetectionRule`/`Finding` analyzer pattern and the
  `securityContext`/`REVIEW REQUIRED` machinery in `DeploymentGenerator`
  already have the exact seams a security-analysis rule set would plug
  into (see `docs/roadmap.md`'s v0.4 section, which already earmarks
  "Security risk detection: privileged mode, host network/PID, dangerous
  capability grants, root filesystem writes").
- It's the most natural next layer of the project's core "never silently
  guess" contract: today, an unresolved UID or a `privileged: true` Compose
  service already produces a Finding/TODO, but there's no dedicated,
  structured policy layer scoring or explaining *why* something is risky —
  that's a coherent, self-contained theme rather than a grab-bag.
- It doesn't require new external integrations (no registry/network access,
  no new input format, no interactive UX design) — lower risk, faster to
  land cleanly, consistent with how v0.1→v0.3 each shipped one cohesive
  capability.
- Profiles (2) and interactive review (4) both are more naturally *later*
  milestones — they benefit from having a richer analysis/security layer to
  build overrides and profiles on top of. Image inspection (3) is already
  explicitly slotted as v0.6 in the roadmap and depends on OCI registry
  access, a heavier and riskier dependency to take on next.
