# Manifest generation (v0.3)

> This project generates maintainable GitOps scaffolding, not automatically
> production-ready Kubernetes deployments. Readability, determinism, safety,
> and human review take priority over generating the fewest lines of YAML.

## Workflow

```
analyze → review → generate → validate
```

```sh
gitops-scaffold analyze docker-compose.yml --format json --output report.json
# review the report, resolve anything you can before generating
gitops-scaffold generate report.json --app myapp --namespace apps --output ./gitops
gitops-scaffold validate ./gitops
```

`generate` also accepts a Compose file directly — both inputs converge on the
exact same pipeline (see "Input resolution" below), so there is only ever one
generation code path, never two.

```sh
gitops-scaffold generate docker-compose.yml --app myapp --namespace apps --output ./gitops
```

## Input resolution

`generate SOURCE` decides how to read `SOURCE` by its extension alone:
`.json` is loaded and trusted as a previously-saved `AnalysisReport`
verbatim — **no re-analysis** happens, which is the entire point of the
cached-report design (an `analyze --output` you've already reviewed doesn't
get silently re-interpreted differently at generate time). Anything else is
parsed and analyzed fresh, via the exact same `detect_parser` + `.parse()` +
`DefaultAnalyzer.analyze()` calls `analyze` itself makes. Both paths produce
one `(ApplicationDefinition, AnalysisResult)` pair before
`GenerationPipeline.generate()` runs — see `generators/pipeline.py`.

**Secret classification is derived from analysis findings, never re-derived
from `looks_like_secret`.** A `report.json` produced by `analyze --output`
has its secret-shaped `EnvVar.value`s already replaced with `"***REDACTED***"`.
If generation re-ran `looks_like_secret` with its own local
`.gitops-scaffold.yaml` (which might configure different secret name
patterns than whatever produced the report), a variable the original
analysis called secret could go unrecognized, and the redaction marker
itself could leak into a ConfigMap as if it were a real value. Instead,
per service, every generator that needs to know "is this variable secret"
scans `AnalysisResult.findings` for `secret-*` codes and reads the variable
name from `Finding.field_path` — fully determined by whatever analysis
already happened, identical whether that analysis is embedded in a cached
report or freshly computed.

## Directory layout

**One service → flat.** Exactly the files below, directly under `--output`.
**Two or more services → one subdirectory per service** (kebab-cased name),
each with its own manifests and `kustomization.yaml`, plus a **root**
`kustomization.yaml`, `README.md`, and `generation-report.json` (never
duplicated per service). A Compose named volume shared by more than one
service gets exactly one PVC, at the root — see "Persistent volumes" below.

```
<output>/
├── configmap.yaml          # when the service has non-secret env vars
├── deployment.yaml
├── service.yaml            # when the service exposes ports
├── pvc.yaml                # when the service needs persistent storage
├── secret.example.yaml     # when secret-shaped env vars were detected
├── ingress.yaml            # only when --ingress-* flags were given
├── kustomization.yaml
├── README.md
└── generation-report.json
```

## Compose → Kubernetes mappings

### command / entrypoint

`ServiceDefinition.entrypoint` → Deployment container `command:`;
`ServiceDefinition.command` → container `args:`. This is the only mapping
that reproduces Docker's actual behavior — verified against all four
combinations, not just the common case:

| Compose | Rendered | Why |
| --- | --- | --- |
| neither set | neither field | image's own ENTRYPOINT + CMD apply |
| only `command` | only `args` | image's ENTRYPOINT + your args |
| only `entrypoint` | only `command` | image's default CMD is **dropped entirely**, not appended — Kubernetes' own docs confirm this is exactly what happens when you set `command` without `args` |
| both | both | straightforward |

### healthcheck → probes

Translated as `exec` commands (a faithful mechanical translation of what
Compose already declared, never a guessed `httpGet`):
`("CMD-SHELL", "cmd")` → `command: ["/bin/sh","-c","cmd"]`; `("CMD", *argv)`
→ `command: argv`; a plain string → `["/bin/sh","-c",string]`.

- **`readinessProbe`** — generated by default whenever a usable healthcheck
  exists.
- **`livenessProbe`** — **not** duplicated automatically. Only added when
  `enable_liveness_probe: true` is configured (see `docs/configuration.md`).
- **`startupProbe`** — generated whenever `start_period` is present, instead
  of folding it into `initialDelaySeconds` (not a faithful translation —
  Compose's `start_period` suppresses failure *counting* during startup;
  `initialDelaySeconds` just delays the first check entirely).
  `periodSeconds` defaults to the healthcheck's own interval, or 10s as a
  fallback if none was declared; `failureThreshold` is computed so the total
  allowed startup window approximates `start_period`.
- **Explicitly disabled** (`test: ["NONE"]` / `disable: true`) is treated
  differently than **absent** — a deliberate opt-out gets a plain
  informational note, not a `TODO`/review-required marker.

**Known limitation:** Compose healthcheck semantics don't map exactly onto
Kubernetes readiness/liveness/startup probes — this is an approximation, not
an equivalence.

### Ports

Only `container_port` ever reaches a Kubernetes object. Compose's
`host_port` has no ClusterIP-Service equivalent at all and is never used as
a `targetPort` or Service `port` — it may appear only in README's
`kubectl port-forward` example (as the arbitrary *local* side, for
familiarity). `targetPort` references the container port's **name** (not a
duplicated number) — the same name the Deployment renders — which
structurally guarantees the two can never drift apart.

### Persistent volumes

**Deliberate v0.2 → v0.3 behavior change:** v0.2's `VolumeDetectionRule`
treats every bind mount as warning-only. v0.3 additionally converts bind
mounts representing real application data into PVC scaffolding (marked
`REVIEW REQUIRED` — the data currently lives on the host and needs
migration), because that's what a real self-hosted app's Compose file
usually looks like (see the Audiobookshelf fixture: `/audiobooks`,
`/podcasts`, `/config`, `/metadata` are all bind mounts). Excluded:

- **Host-system paths** (`/proc`, `/sys`, `/dev`, `/var/run/docker.sock`,
  `/run/docker.sock` prefixes) — always skipped, never a PVC.
- **Known single-file mounts** (`/etc/localtime`, `/etc/timezone`,
  `/etc/hosts`, `/etc/resolv.conf`) — always skipped.
- **Ambiguous file-looking mounts** — read-only, with a target whose last
  path segment has a `.` (looks like a filename with an extension, e.g.
  `nginx.conf`) — skipped with a `REVIEW REQUIRED` note suggesting a
  ConfigMap/Secret volume might fit better. This is a permanent limitation:
  there's no way to inspect the real host filesystem to know for certain
  whether a bind mount target is a file or a directory.
- **tmpfs** — never converted (ephemeral by definition). Not wired to
  `emptyDir` either, to keep scope controlled — a safe future improvement,
  not implemented now.
- **Anonymous volumes** (no name) — converted, named from the slugified
  target path, marked `REVIEW REQUIRED` (intent unclear without a name).

**Named-volume identity:** a Compose named volume is identified by its
`source` string and can be mounted by more than one service. PVC generation
collects named volumes once per unique `source` across the whole app —
services sharing one Compose volume share one PVC (named `kebab(source)`,
at the root), not one each.

Every PVC's size is `default_pvc_size` (default `1Gi`) and is **always**
marked `REVIEW REQUIRED` — Compose has no size concept at all, full stop.

**Known cosmetic overlap:** a PVC derived from a bind mount at `/config`
and the service's ConfigMap both end up named `<service>-config` — this is
harmless (Kubernetes scopes names per-kind, so a ConfigMap and a PVC with
the same name in the same namespace don't conflict), but can read as
confusing side-by-side in `kubectl get` output.

### Resources

CPU passes through as-is (decimal cores are valid in both Compose and
Kubernetes). Memory strings are checked against a permissive
Kubernetes-quantity shape (`512M`, `1Gi`, `256K`, ...); a value that doesn't
match is flagged `REVIEW REQUIRED` rather than silently converted, since
guessing the wrong unit would be worse than asking. Configured defaults
(`default_cpu_request`, etc.) are injected only when Compose provided
nothing for that specific field.

### Runtime user / securityContext

`securityContext` is only rendered when a numeric UID was actually resolved
from Compose's `user:` field. Unresolved (no `user:` at all, or a
by-name user like `user: appuser` that can't be resolved to a UID without
inspecting the image) → a `REVIEW REQUIRED` comment, never an invented
value.

### Secrets

Secret-shaped variables never appear as plaintext anywhere generated —
`secret.example.yaml` (generated only when at least one exists) always uses
the fixed placeholder `CHANGE_ME`, and the Deployment references them via
`secretKeyRef` pointing at a Secret that doesn't exist yet. `optional` on
each `secretKeyRef` is set only for variables sourced entirely from the
shell environment (Compose's closest thing to "optional") — a literal
value, an interpolation, or an explicit empty string are all treated as
required (Kubernetes' own default).

**The Deployment will not become Ready until the real Secret exists** —
README.md and `generation-report.json` both state this explicitly, and name
the exact live Secret name plus its required/optional keys, with a
ready-to-edit `kubectl create secret generic ...` command.

### Restart policy

Compose's `restart:` is never rendered as any Deployment field — there's no
direct equivalent (Deployments always maintain the desired replica count
regardless of it). A note documents the mismatch instead.

## Overwrite safety

By default, `generate` refuses to touch any file it isn't certain it created
itself. It uses the **previous run's own `generation-report.json`** (if one
exists in the output directory) as the ledger of "what did gitops-scaffold
generate here last time":

1. **Foreign files** — exist on disk, but weren't recorded as ours by a
   previous run (or no previous run exists at all). **Always blocks, even
   with `--force`** — this is the actual meaning of "only replace files
   managed by gitops-scaffold."
2. **Managed conflicts** — our own prior output, at a path being
   regenerated. Blocks without `--force`; overwritten with it (recorded in
   `overwritten_files`).
3. **Orphaned files** — generated by a previous run but not produced by
   this one (a removed service, or an app crossing the 1↔2+ service layout
   boundary). Blocks without `--force`; with it, **left in place, never
   deleted**, and recorded in `orphaned_files` with a loud warning.

All checks run before anything is written — a blocked run touches nothing.

## Ingress

Off by default. Generated only when **all four** of `--ingress-host`,
`--ingress-class`, `--tls-secret`, and `--cluster-issuer` are given together
— a partial combination is a hard CLI error, not a partially-defaulted
Ingress. Applies to the first service (in declaration order) that has a
Service generated; with more than one candidate, only the first gets an
Ingress and a note explains the choice. No Cloudflare Tunnel configuration.

## Validation

`gitops-scaffold validate <dir>` runs `StructureValidator` (expected files
present, no unresolved `TODO`/`REVIEW REQUIRED` markers) and
`ManifestConsistencyValidator` (every YAML parses; `kustomization.yaml`
resources exist and never include the three forbidden files; Service
selectors match Deployment pod labels; Service `targetPort`s match named
container ports; `volumeMounts` match declared volumes; PVC `claimName`
references exist; `secret.example.yaml` values are all exactly
`CHANGE_ME`; the redaction marker never appears in any manifest; resource
names are valid Kubernetes names). `--kubectl` additionally runs
`kubectl kustomize <dir>` if `kubectl` is found on `PATH` — never a hard
dependency; its absence is informational, not a failure.

## Upgrade / rollback expectations

Re-running `generate` over a previous output is expected and safe (that's
what the overwrite ledger is for). There is no automatic rollback — Git is
the rollback mechanism: commit generated output before applying it, and
`git revert` if a generation turns out wrong. Changing an app's service
count (1 ↔ 2+) changes the whole directory shape; the orphan-detection
mechanism above will tell you exactly what's stale rather than silently
leaving it behind un-flagged.

## Unsupported / out of scope for v0.3

No Helm generation. No OpenBao/ExternalSecret generation. No Cloudflare
Tunnel. No live `kubectl apply`. No Flux reconciliation. No `build:`-context
support (a build-only service is skipped entirely, with a note). Port
ranges aren't modeled (recorded as unsupported fields at the parser level,
per `docs/compose-support.md`).
